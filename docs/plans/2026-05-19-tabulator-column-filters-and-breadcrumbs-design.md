# Tabulator Column Filters and Breadcrumb Navigation Design

## Goal

Refine the new server-first list experience so it feels more like a self-contained data grid and less like a page with a separate filter form. The new UX should remove the top query-builder form, move sorting and filtering into the table columns via Tabulator, add lightweight breadcrumb navigation across the app, and keep a clear loading indicator in the table area during server-driven navigation.

## Validated decisions

- Remove the top query/filter form from list pages.
- Use Tabulator column interactions for list UX.
- Prefer Tabulator-provided interactions where available.
- Add breadcrumb navigation rather than a heavy global nav bar.
- Keep the ASP.NET list endpoint authoritative for filtering, sorting, and row rendering.
- Preserve the current URL contract (`filter.N.*`, `sort.N.*`, `offset`, `limit`) so links stay shareable and related-row navigation keeps working.
- Keep the HTML fallback table, FK links, and row actions intact when enhancement is unavailable.

## Recommended approach

Use a hybrid model where **Tabulator owns the list interaction surface** and **the server owns the semantics and rendered result set**.

That means:

- Tabulator provides sortable headers, header filters, and grid-local loading UI.
- Client interactions are translated into the existing server URL query contract.
- The browser then navigates to the resulting URL.
- The server applies the query, renders the authoritative list HTML, and emits the next enhancement payload.

This preserves the server-first model, keeps URLs stable, avoids introducing a separate JSON API in this pass, and removes the visual redundancy of a top-of-page query form.

## Page structure

### All pages

Every EF UI page should render a lightweight breadcrumb bar near the top of the main content.

Examples:

- `EF UI`
- `EF UI / Chinook`
- `EF UI / Chinook / Tracks`
- `EF UI / Chinook / Tracks / Edit`

The breadcrumb should be fully server-rendered and link back to the relevant higher-level pages. This provides a reliable way to get back to the root or mount index even if client enhancement fails.

### List pages

List pages should render, in order:

1. breadcrumb bar
2. page title
3. primary action area (`Create New`)
4. table enhancement host
5. fallback HTML table
6. lightweight table status/error area when needed

The current top query-builder section should be removed completely from the visible page layout.

## Table interaction model

Tabulator should become the visible interaction layer for list pages.

### Sorting

- Sortable columns use Tabulator header sorting.
- Clicking a sortable header updates `sort.0.field` and `sort.0.dir` in the URL.
- Re-click behavior should rotate through ascending, descending, and cleared sort if Tabulator’s behavior allows it cleanly.
- The actions column must never be sortable.

### Filtering

- Filterable columns use Tabulator header filters.
- Header filter changes update the canonical query contract:
  - `filter.0.field`
  - `filter.0.op`
  - `filter.0.value`
- The first pass should stay intentionally small and predictable:
  - one active sort
  - one active filter per interaction cycle unless multi-filter support is added explicitly
- The actions column must never be filterable.

### FK column behavior

Foreign-key columns should still display their label text in the grid, but filtering semantics must remain server-defined.

Rules:

- `contains` should keep working against the displayed label semantics.
- `eq` must continue to accept raw FK values when those URLs are generated by related-row navigation or pasted directly.
- FK cells continue linking to related edit pages.

This preserves both user-friendly label display and robust direct URL behavior.

## Enhancement payload changes

The server-rendered enhancement config should become slightly richer so the client can build the correct column UX without inventing rules.

Each column definition should carry at least:

- `field`
- `title`
- `sortable`
- `filterable`
- default or allowed filter operator behavior
- active sort state
- active filter state
- whether the column is the synthetic actions column

Each row should continue carrying:

- rendered cell text
- optional cell href
- rendered row actions markup for the actions column

The client should consume this metadata to build Tabulator columns consistently with the server’s capabilities.

## Loading behavior

Use a **grid-local loading indicator** rather than a detached page banner.

Preferred behavior:

- show loading immediately when the user changes a header filter or sort
- keep the indicator visually attached to the table area
- use Tabulator’s loading/processing presentation where practical
- navigate to the computed server URL after entering the loading state

This keeps feedback close to the interaction and avoids the awkward feeling of a top-of-page form submitting separately from the table.

## Error handling and fallback

The server should remain responsible for validation and query safety.

If the URL contains invalid query parts:

- the server should reject them gracefully
- the page should still render
- the user should see a compact status/error area near the table instead of the removed top form

If enhancement fails entirely:

- breadcrumbs still work
- page title and create action still work
- fallback HTML table still renders
- row actions still work
- FK links still work

Only the in-column sorting/filtering experience is lost.

## Navigation design

Breadcrumbs should be generated from current route context.

Suggested breadcrumb model:

- root crumb: EF UI home
- mount crumb: current mount root (for example `/chinook`)
- entity crumb: current entity list
- page-mode crumb when relevant (`New`, `Edit`)

This should be lightweight, server-rendered, and reused by index, list, and form pages.

## Testing strategy

### Renderer tests

Add/adjust renderer tests to verify:

- breadcrumb markup is present on index, list, and form pages
- the top query form is no longer rendered on list pages
- the enhancement config contains sortable/filterable column metadata
- the synthetic actions column remains present in enhancement config
- the fallback HTML table still includes edit/delete actions

### Endpoint tests

Add/adjust endpoint tests to verify:

- breadcrumb links render on `/simple` and `/chinook` pages
- list pages no longer render the old top query form
- header-filter/sort state is represented in the enhancement config
- FK filter URLs like `MediaTypeId eq 1` still return rows
- related-row links still produce visible, valid filtered child lists

### Asset tests

Add/adjust asset tests to verify:

- Tabulator enhancement script wires header sorting
- Tabulator enhancement script wires header filters
- grid-local loading state is shown during navigation
- actions column is excluded from sorting/filtering behavior

## Non-goals for this pass

- full client-side data ownership
- introducing a dedicated remote JSON table API
- multi-column sort orchestration beyond the current canonical contract
- arbitrary nested relationship filtering
- replacing the fallback HTML-first strategy

## Implementation note

This design should be implemented in an isolated worktree before touching production behavior again.
